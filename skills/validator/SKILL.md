# 工作流 (Workflow)

1. 读取 IR 主文档、追踪矩阵、验收标准、待确认问题和上游专家产物，重点对照 `ambiguity-map.md`、`decision-records.md` 和 `requirement-maturity-assessment.md`。
2. 检查 RR 覆盖、竞品参考处理、专家产物一致性和追踪完整性。
3. 检查验收标准是否具体、可验证、可关联需求。
4. 输出阻塞问题、警告、通过项和建议修复动作。

# Tool Usage Notes

- 可以使用 `validate_artifacts` 做结构化校验，但最终判断要结合文本证据。
- 不重写 IR，也不新增需求。
- 每个问题必须定位到文件、条款或证据来源。
- 必须检查“待确认”项是否被错误写成已确认需求，以及“按假设推进”项是否保留假设来源和适用边界。

# 最终生成策略 (Final Generation)

1. 验证报告按阻塞、警告、通过项、建议动作分类。
2. 每条问题说明影响、证据、建议修复和是否阻塞进入 IT 设计。
3. 验证报告必须包含澄清状态闭环检查，覆盖已确认、待确认、按假设推进和非本轮范围。
4. 所有自然语言默认使用简体中文。
